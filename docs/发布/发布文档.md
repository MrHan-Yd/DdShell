# DdShell 发布指南

## 发布流程

DdShell 使用 GitHub Actions 自动构建三端安装包并发布到 GitHub Releases。

应用内更新使用 Tauri 官方 updater。macOS 和 Windows 走签名校验后的应用内下载、安装与用户确认重启；Linux 暂时保留 GitHub Releases 手动下载 fallback。

### 触发方式

推送以 `v` 开头的 tag 即可触发：

```bash
git tag v0.1.0
git push origin v0.1.0
```

### 自动构建的平台和产物

| 平台 | Runner | 产物文件名 | 说明 |
|------|--------|-----------|------|
| macOS (Apple Silicon) | `macos-latest` | `DdShell-v0.1.0-macos-aarch64.dmg` | M1/M2/M3/M4 芯片 |
| macOS (Intel) | `macos-latest` | `DdShell-v0.1.0-macos-x86_64.dmg` | Intel 芯片 |
| Windows | `windows-latest` | `DdShell-v0.1.0-windows-x64.msi` | 64位 MSI 安装包 |
| Windows | `windows-latest` | `DdShell-v0.1.0-windows-x64.exe` | 64位 NSIS 安装包 |
| Linux | `ubuntu-22.04` | `DdShell-v0.1.0-linux-amd64.deb` | Debian/Ubuntu |
| Linux | `ubuntu-22.04` | `DdShell-v0.1.0-linux-amd64.AppImage` | 通用 Linux |

Tauri 官方 updater 还会发布以下更新产物：

| 平台 | 产物文件名 | 用途 |
|------|-----------|------|
| macOS (Apple Silicon) | `DdShell-v0.1.0-macos-aarch64.app.tar.gz` + `.sig` | 官方 updater 下载与签名校验 |
| macOS (Intel) | `DdShell-v0.1.0-macos-x86_64.app.tar.gz` + `.sig` | 官方 updater 下载与签名校验 |
| Windows | `DdShell-v0.1.0-windows-x64.exe` + `.sig` | 官方 updater 下载与签名校验，NSIS 安装器本身可作为更新包 |
| 全平台 | `latest.json` | updater manifest，指向当前最新版本的更新产物 |

### Workflow 文件

`.github/workflows/release.yml`

#### 流程：

1. **create-release** — 创建 GitHub Release（名称 `DdShell-v0.1.0`），写入下载说明
2. **build**（4 个并行 job）— 各平台编译 Tauri 应用
3. 构建完成后重命名产物为统一格式 `DdShell-{tag}-{platform}.{ext}`
4. 上传普通安装包与 updater 产物
5. **publish-updater-manifest** — 读取 `.sig`，生成并上传 `latest.json`

### Tauri updater 签名

官方 updater 使用独立的 minisign 签名密钥，不等同于 macOS Developer ID 签名或 Windows 代码签名。

仓库中的 `app/src-tauri/tauri.conf.json` 保存 updater 公钥；GitHub Actions 使用私钥给更新产物签名。

GitHub 仓库需要预先配置以下 Secrets：

| Secret | 用途 |
|------|------|
| `TAURI_SIGNING_PRIVATE_KEY` | Tauri updater 私钥内容 |
| `TAURI_SIGNING_PRIVATE_KEY_PASSWORD` | 私钥密码 |

生成 updater 签名密钥示例：

```bash
cd app
pnpm tauri signer generate --password "<strong-password>" --write-keys ~/.tauri/ddshell-updater.key
```

命令会输出：

- 私钥文件：只放本机安全位置或 GitHub Secrets，不能提交
- 公钥：写入 `app/src-tauri/tauri.conf.json` 的 `plugins.updater.pubkey`

如果更换 updater 私钥，必须同步更新应用内置公钥；旧版本客户端无法验证新私钥签出的更新包。

### macOS 签名与公证

当前 macOS Release 有两种模式：

- 配置了 Apple Secrets：执行 Developer ID Application 签名与 Apple notarization 公证
- 未配置 Apple Secrets：执行完整 `adhoc` 重签名，避免产出会被系统直接判为“已损坏”的 `.app`

GitHub 仓库需要预先配置以下 Secrets：

| Secret | 用途 |
|------|------|
| `APPLE_CERTIFICATE` | Base64 编码后的 Developer ID Application `.p12` 证书 |
| `APPLE_CERTIFICATE_PASSWORD` | `.p12` 证书密码 |
| `APPLE_SIGNING_IDENTITY` | 签名身份，例如 `Developer ID Application: Your Name (TEAMID)` |
| `APPLE_ID` | Apple Developer 账号邮箱 |
| `APPLE_PASSWORD` | Apple ID 的 app-specific password |
| `APPLE_TEAM_ID` | Apple Developer Team ID |

准备 `APPLE_CERTIFICATE` 的示例命令：

```bash
base64 -i developer-id-application.p12 | pbcopy
```

如果这些 Secrets 缺失，workflow 不会中断，而是退回到 `adhoc` 重签名模式。

这种模式下的 macOS 包通常表现为：

- 可以安装和运行
- 可能会提示“无法验证开发者”或需要用户在“隐私与安全性”中手动放行
- 不应再出现由错误 bundle 签名导致的“App 已损坏”

## 发布新版本步骤

### 1. 本地检查（必做）

在推送代码前，先在本地运行构建检查，确保没有错误：

```bash
cd app
pnpm build
```

如果构建成功，再进行下一步。

### 2. 更新版本号

同时修改以下三处：

```bash
# app/src-tauri/tauri.conf.json
"version": "0.2.0"

# app/src-tauri/Cargo.toml
version = "0.2.0"

# app/package.json
"version": "0.2.0"
```

同时更新 `.github/workflows/release.yml` 里的 GitHub Release 正文版本介绍。

位置在 `create-release` job 的 `body` 字段中，放在：

```md
## DdShell ${{ github.ref_name }}
```

和：

```md
开源跨平台 SSH/SFTP 桌面客户端。
```

之间。每次发布新版本时，只替换这一段版本介绍，不要保留上一版文案；`开源跨平台 SSH/SFTP 桌面客户端。` 以及它下面的下载表格、macOS 用户须知等固定发布说明继续保留。

示例：

```yaml
body: |
  ## DdShell ${{ github.ref_name }}

  本次版本重点改进了应用内更新体验：macOS 和 Windows 现在通过 Tauri 官方 updater 完成签名校验、应用内下载和安装，并在安装完成后提示用户确认重启。

  开源跨平台 SSH/SFTP 桌面客户端。
```

### 3. 提交并打 tag

```bash
git add -A
git commit -m "release: v0.2.0"
git tag v0.2.0
git push origin main --tags
```

### 4. 等待构建并验证 Release

- 查看进度：https://github.com/MrHan-Yd/DdShell/actions
- 首次构建约 10-15 分钟（有 Rust 缓存后约 5-8 分钟）
- 打开 https://github.com/MrHan-Yd/DdShell/releases
- 确认 6 个安装包都已上传
- 确认 macOS / Windows updater 产物和 `.sig` 都已上传，其中 Windows updater 签名应为 `DdShell-{tag}-windows-x64.exe.sig`
- 确认 `latest.json` 已上传且内容包含 `darwin-aarch64`、`darwin-x86_64`、`windows-x86_64-nsis`、`windows-x86_64`
- 检查下载链接是否可用
- 用旧版本应用执行“检查更新 → 下载并安装 → 提示重启”，确认不会自动重启

### 5. 生成版本介绍文案

每次发布都需要根据本次版本的实际更新内容生成版本介绍文案，可用于 GitHub Release、README 更新、社交平台或版本介绍。用于 GitHub Release 的版本介绍必须在打 tag 前填入 `.github/workflows/release.yml` 的 `body` 顶部，位置见上面的“更新版本号”步骤。

生成前先确认本次发布范围：

```bash
git log <上一版本tag>..HEAD --oneline
git diff --stat <上一版本tag>..HEAD
```

版本介绍应包含：

- 版本标题：`DdShell vX.Y.Z 发布`
- 一段概述：说明本次版本的核心目标和用户价值
- 亮点：列出最重要的新功能或体验改进
- 其他改进：列出次要优化、文档、兼容性或稳定性改动
- 已知限制：如有影响用户使用的限制，需要明确写出

推荐模板：

```md
# DdShell vX.Y.Z 发布

<用 1 段话说明本次版本的核心变化、解决的问题和适合用户关注的价值。>

## 亮点

### <亮点 1>

<说明用户能做什么、体验有什么提升。>

### <亮点 2>

<说明用户能做什么、体验有什么提升。>

### <亮点 3>

<说明用户能做什么、体验有什么提升。>

## 其他改进

- <改进 1>
- <改进 2>
- <改进 3>

## 已知限制

- <如无明显限制，可省略本节。>
```

## 本机打包（仅 macOS）

如需在本机测试打包：

```bash
cd app
export TAURI_SIGNING_PRIVATE_KEY="$(cat ~/.tauri/ddshell-updater.key)"
export TAURI_SIGNING_PRIVATE_KEY_PASSWORD="<private-key-password>"
pnpm tauri build
```

产物位置：
- `app/src-tauri/target/release/bundle/macos/DdShell.app`
- `app/src-tauri/target/release/bundle/dmg/DdShell_0.1.0_aarch64.dmg`
- `app/src-tauri/target/release/bundle/macos/DdShell.app.tar.gz`
- `app/src-tauri/target/release/bundle/macos/DdShell.app.tar.gz.sig`

## 重新发布（删除旧 Release）

```bash
# 删除本地和远程 tag
git tag -d v0.1.0
git push origin :refs/tags/v0.1.0

# 删除 GitHub Release（需要 gh CLI）
gh release delete v0.1.0 --repo MrHan-Yd/DdShell --yes

# 重新打 tag 推送
git tag v0.1.0
git push origin v0.1.0
```

## macOS 安装说明

打开 `.dmg` 后看到两个图标是正常的：

- `DdShell.app`：应用本体
- 灰色 `Applications`：系统应用目录的快捷入口

正确安装方式：

1. 将 `DdShell.app` 拖到灰色 `Applications`
2. 等复制完成
3. 从“应用程序”中启动 `DdShell`

如果用户下载了正确架构的包，但仍出现“App 已损坏”，通常说明发布包的 bundle 签名状态异常，而不是 DMG 展示方式有问题。
