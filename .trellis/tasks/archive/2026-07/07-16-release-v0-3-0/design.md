# v0.3.0 发布设计

## 发布边界

发布准备只修改版本元数据、Release 文案、README 最新版本链接和新增版本介绍，不改变应用业务逻辑。正式分发由现有 `.github/workflows/release.yml` 负责。

## 发布数据流

本地发布提交 → 推送 `main` → 创建并推送 `v0.3.0` → GitHub `create-release` → macOS/Windows/Linux 并行构建 → 上传安装包和签名 → 生成并上传 `latest.json` → 旧客户端通过固定 updater URL 检查更新。

## 版本介绍结构

- 核心定位：主题系统正式扩展，提供 16 套可选主题。
- 亮点一：14 套新增主题，覆盖多种材质和配色方向。
- 亮点二：主题与深色/浅色/系统模式分离，设置中支持预览、保存与重启恢复。
- 亮点三：设置、连接、终端、SFTP 和应用壳完整适配，不是单纯换色。
- 其他改进：主题命名与选中样式统一、雨湖斜线纹理、连接长名称截断。

## 风险与控制

- 本地分支领先远端较多：发布前先推送 main，再推送标签，确保标签所指提交在远端可达。
- 发布是不可逆外部操作：推送前验证版本号、文案、构建和 tag 不存在；失败时不移动现有 tag，先根据 workflow 日志修复并按重发文档处理。
- updater 依赖签名 Secrets：发布前使用 `gh secret list` 确认两个 Tauri Secrets 存在。
- macOS 无 Developer ID：版本说明保留系统放行提示，不宣称已公证。

## 回滚

若标签尚未推送，可撤销发布准备提交后修复；若标签已推送并创建 Release，按发布文档删除远端 Release/tag，修复后重新创建同版本标签。
